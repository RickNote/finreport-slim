# finreport-slim 开发笔记

供后续AI会话参考的修改历史、公司特异性问题、架构决策和踩坑记录。

## 适用范围

聚焦A股上市保险公司三类定期报告：
- **年报**（年度报告）→ 按章节提取（section-slim）
- **中报**（中期报告/半年度报告）→ 按章节提取（结构与年报基本一致）
- **偿付能力报告** → 按主题关键词筛选（extract-theme，theme=solvency）

根据PDF文件名判断类型。

## 架构演进

### v1: 关键词打分筛页（extract-theme + slim）

原始设计。按主题关键词给每页打分，超过阈值的页保留。适合模糊主题（如"投资收益"），但无法精确定位固定章节。**目前仍保留**，用于自定义主题提取。

### v2: TOC驱动定位（toc-scan + section-slim）

2026-04引入。从年报目录解析页码范围，精确定位4个固定章节。不需要关键词。解决了关键词方法"噪声多、不精确"的问题。

### v3: 附注科目筛选（--ref-slim）

同期引入。从已提取的资产负债表/利润表中解析"附注X"编号，只提取对应子项。将附注从100+页缩减到40-60页。

### 废弃功能

`locate` 和 `build-records` 命令已删除。它们试图做行级附注映射（主表行→附注段落），但实现复杂且不稳定。被 `section-slim --ref-slim` 的页级筛选取代。

## 公司特异性问题

### 目录格式差异

| 公司 | 目录格式 | 偏移量 |
|------|---------|--------|
| 中国平安 | HTML表格 `<table><tr><td>标题</td><td>页码</td>` | 从锚点计算 |
| 新华保险 | 正向文本 `标题 页码` | 从锚点计算 |
| 中国太保 | **反向文本** `页码 标题`（如 `17 经营业绩回顾与分析`） | 实际偏移16，但锚点算出0（见下方bug） |
| 中国人保 | 正向文本 | 偏移66（H股前言部分） |
| 中国人寿 | 正向文本 | 偏移约28-29（按年份变化） |

### H+A双重上市的偏移量问题（人保、人寿）

**核心问题**：偏移量从"合并资产负债表"锚点计算，只对财务报表区域正确。年报前半部分（H股格式）使用不同的页码体系。

- 人保：管理层讨论与分析在 page_idx=19（H股区域），但TOC用偏移66映射到 page_idx=85（A股区域的董事会报告）
- 人寿：同理，偏移28把管理层分析映射到了公司治理区域

**解决方案**：TOC命中后验证。用 `_matches_as_heading()` 检查目标页第一行是否确实包含该章节的body_pattern。验证失败则回退到正文扫描。

### 太保偏移量为0的问题

太保TOC中没有"合并资产负债表"等锚点条目（财务报表在子目录中），导致 `_compute_toc_offset` 算出偏移0。实际偏移应为16（16页前导图文）。

**影响**：management-discussion的TOC匹配指向错误位置。被验证机制捕获并回退到body scan。

### 各公司管理层分析叫法

| 公司 | 标题 |
|------|------|
| 新华保险 | 管理层讨论与分析 |
| 中国人寿 | 管理层讨论与分析 |
| 中国人保 | 管理层讨论与分析 |
| 中国平安 | 业绩综述 |
| 中国太保 | 经营业绩回顾与分析 |

### 财务子目录（Sub-TOC）

人保和太保在"财务报告"章节有内嵌子目录，列出各报表页码：
- 人保（page 119）：绝对页码格式 `合并及公司资产负债表 126`
- 太保（page 152）：P前缀相对页码 `合并资产负债表 P5`（P5 = 子目录页+5）

`_extract_financial_page_map()` 自动检测并处理两种格式。

### 中国人寿 2025 的财务报表特殊点

- TOC 只有 `08 财务报告`，没有单列 `合并资产负债表` / `合并利润表` / `财务报表附注`
- 合并资产负债表第一页直接以表格开头，表头类似 `资产 / 附注十 / 2025年12月31日 / 2024年12月31日`
- 因此三大主表主要依赖 body scan，而不是 TOC 精确命中

### 附注格式差异

| 公司 | 表头引用格式 | 附注编号 | 主节 |
|------|------------|---------|------|
| 中国太保 | `附注六` + 子编号 1,2,3... | `六、合并财务报表主要项目附注` | 六 |
| 中国人保 | `附注七` + 子编号 | `七、合并财务报表项目附注` | 七 |
| 中国平安 | `附注八` + 子编号 | `八、合并财务报表项目附注` | 八 |
| 中国人寿 | `附注十一` + 子编号 | `十一、合并财务报表项目附注` | 十一 |
| 新华保险 | 无主节前缀，直接 `8/56(1)` | 阿拉伯数字顶级编号 `8 货币资金` | 无 |

**注意**：这些编号不是固定的，换一个年份可能会变。代码从当年实际报表中动态解析。

## 已解决的关键Bug

### 1. 审计报告误杀子目录页

**现象**：人保balance-sheet定位失败。
**原因**：`_extract_financial_page_map()` 中的审计报告过滤器检查页面前150字符是否含"审计报告"。人保page 119（财务子目录）列有"一、审计报告 120"，被误判跳过。
**修复**：移除子目录扫描中的审计报告过滤。`len(found) >= 2` 条件本身已防止误匹配。

### 2. 目录页和mini-TOC误触发正文扫描

**现象**：太保管理层分析回退到body scan后，匹配到page 14（目录页）或page 30（mini-TOC）。
**原因**：`_matches_near_top()` 检查前220字符，目录页和mini-TOC中的章节名也在范围内。
**修复**：引入 `_matches_as_heading()`，只检查第一行，并跳过包含2+行"标题 数字"格式的页面（mini-TOC检测）。

### 3. 表格单元格中的stop关键词误触停止

**现象**：太保management-discussion只提取4页（33-36），应该到page 62。
**原因**：page 37有一个表格包含"集团内含价值"，stop pattern "内含价值"在前220字符内匹配。
**修复**：非财务section的stop扫描改用 `_matches_as_heading()`（只检查第一行），避免匹配表格内容。

### 4. merge_consecutive段落范围过窄

**现象**：人保management-discussion只提取3页（85-87）。
**原因**：TOC中"3. 管理层讨论与分析"和"公司业务概要"共享同一reported_page_no，导致end < start。修复逻辑找到了第一个start > current的条目，但那只是一个子条目。
**修复**：对merge_consecutive类型，先在后续TOC条目中用stop_body_patterns查找下一个顶级节，以此确定段落终止位置。

### 5. 人寿附注包含"公司口径"

**现象**：人寿附注提取包含"十二、公司财务报表主要项目注释"。
**原因**：stop检测只看 `text[:30]`，但人寿每页第一行是运行页眉"2024年度财务报表附注（续）"，真正的节标题在第二行。
**修复**：stop检测改为检查前3行（`"\n".join(text.split("\n")[:3])`）。

### 6. 新华附注缺少notes 8-9

**现象**：新华附注从page 190（note 10）开始，缺少pages 188-189（note 8-9的数据）。
**原因**：pages 188-189以 `<table>` 开头，没有"8."或"9."编号标题。之前的note tracking停留在section 5。
**修复**：无主节前缀模式下，当current_subnote < first_bs_note 且页面以 `<table>` 开头时，假设进入了第一个被引用的附注。

### 7. `toc-scan` / notes 运行时缺少 `_extract_tables_from_text`

**现象**：`toc-scan` 和部分附注辅助路径在运行时直接 `NameError`。
**原因**：`finreport_scope.py` 中调用了 `_extract_tables_from_text()`，但函数定义缺失。
**修复**：补回通用 HTML 表格解析 helper，并统一给 TOC 扫描、附注辅助提取和 Markdown 表格转换复用。

### 8. MinerU HTML 表格的空表头和空标签汇总行噪声过多

**现象**：平安、人保、太保、新华的 `slim` 文件中存在大量 `|  | ... |` 行，降低 LLM 可读性。
**原因**：MinerU 表格常把左上角占位单元格、空标签合计行和重复汇总行直接输出为空单元格。
**修复**：
- 空左上角表头在大多数财务表中补成 `项目`
- 多层表头压平成组合列名，如 `2025年度 / 年初余额`
- 空标签但值列明确的汇总行补成 `合计`
- 与前文数值完全重复的空标签合计行直接去重

### 9. 中国人寿 2025 的资产负债表无法定位

**现象**：`section-slim --section balance-sheet` 在中国人寿 2025 年报上找不到起始页。
**原因**：正文第一页没有 `合并资产负债表` 标题，原 `body_patterns` 只覆盖“显式标题”或“资产 … 附注 … 合并”。
**修复**：为 `balance-sheet` 增加更宽的表头模式，支持 `资产 / 附注十 / 2025年12月31日 / 2024年12月31日` 这类直接起表格式。

## 附注筛选机制

### 引用解析（_parse_note_refs_from_slim）

1. 从slim文件提取主节编号（"附注六"→六）和子项编号（表格中的 `| 1 |`、`| 8/56(1) |`）
2. 无主节编号时（新华），用第一个BS科目的编号作为阈值，过滤掉会计政策等通用节

### 页面追踪（_filter_notes_by_refs）

逐页扫描附注区域，追踪当前活跃的子项编号：
1. 阿拉伯数字标题 `N. xxx`
2. 续页 `N. xxx（续）`
3. 中文数字节标题（新华格式）
4. 无标题但以 `<table>` 开头（fallback，见上方bug 6）

只保留编号在引用集合中的页面。

## 已知局限性

1. **偏移量计算依赖锚点**：如果年报中"合并资产负债表"等锚点不在TOC中，偏移量可能不准。验证+fallback机制可兜底。
2. **新华notes 8-9的fallback**：依赖"页面以 `<table>` 开头"的启发式规则，如果MinerU提取格式变化可能失效。
3. **附注编号提取用正则**：表格中的数字可能有假阳性（如日期 `2025年12月31日` 中的数字）。目前通过 `\d{1,2}` + 位置过滤控制。
4. **MinerU解析质量**：脚本依赖MinerU的解析结果，如果PDF解析质量差（如扫描版），需要用 `--ocr` 参数。
5. **OCR/抽取残留连字**：个别页仍会出现 MinerU 原始脏文本，例如数字与页码粘连、全角空格异常等，当前仅做轻量清洗，未做 OCR 纠错。

## 未来可能需要调整的点

1. **年报格式变更**：每年新报告发布后，需验证toc_patterns和body_patterns是否仍然匹配。重点关注：
   - 章节标题措辞变化
   - 附注编号体系变化（如从"附注六"变成"附注七"）
   - 目录格式变化

2. **新公司支持**：添加新保险公司时，需检查：
   - 目录格式是否为已支持的三种之一
   - 是否有H+A双重上市的偏移量问题
   - 附注引用格式

3. **sections.py 更新**：如果某公司的章节名称变化，在 `toc_patterns` 和 `body_patterns` 中添加新模式。模式按优先级排列。

4. **themes.py 更新**：自定义主题的关键词可能需要根据新准则调整（如IFRS 17影响保险合同相关术语）。
5. **公共 helper 抽模块**：`convert.py` 与 `finreport_scope.py` 仍存在部分 MinerU / page-grouping helper 重复，后续可抽成共享模块，降低漂移风险。
